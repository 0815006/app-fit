
健身打卡榜单的功能，其核心价值在于“社交驱动力”与“数据透明度”。在健身领域，榜单不仅是简单的排名，更是用户自我效能感的来源。

针对你的小程序，我们将榜单功能拆解为**逻辑架构、数据颗粒度、激励机制与防作弊**四个维度进行分析。

---

### 一、 榜单的维度设计（由易到难）

榜单如果只有一种“总排行”，会迅速失去吸引力（大神永远霸榜，新手永远垫底）。建议采用“混合榜单模型”：

1. **勤奋榜（输入维度）**：

* **打卡天数榜**：本月/本周累计打卡天数（最基础，鼓励坚持）。
* **动作总量榜**：本月累计完成的训练动作组数（鼓励量变）。

2. **硬核榜（输出维度）**：

* **重量巅峰榜**：各肌群下的历史最大重量（如：卧推 100kg）。
* **进步百分比榜**（最具激励性）：相比上个月，个人力量提升幅度最大的用户（哪怕只有 5kg 到 10kg，对新手也极具成就感）。

3. **恢复榜（反向维度）**：

* **科学训练榜**：根据你已有的“恢复倒计时”逻辑，评选出“恢复期执行率最高者”（即：不盲目乱练，严格遵循肌肉恢复周期的用户）。

### 三、 技术实现的难点与分析

#### 1. 数据的时效性与存储（冷热数据分离）

* **难点**：如果每次查看榜单都从 `logs` 表中进行 `COUNT` 或 `SUM` 实时计算，当用户量大时，数据库压力会非常大。
* **分析**：建议采用“看板存储”。每天凌晨或每小时进行一次聚合计算，将统计结果存入 `rankings` 表中，前端直接读取 `rankings` 表进行展示，从而保证秒开。

#### 2. “含金量”的算法优化

* **难点**：单纯的“动作总数”很容易被恶意刷榜（比如重复录入空组）。
* **分析**：引入“有效训练分”。例如：
* `训练得分 = (训练天数 * 权重) + (力竭度总分 * 权重) + (动作覆盖肌群多样性 * 权重)`。
* 这样能保证那些科学训练、全面均衡发展的用户，排在只练单一动作的“刷榜机器”前面。

#### 3. 社交关系的闭环

* **核心逻辑**：小程序榜单最好的形式是“好友圈榜单”优先，全网榜单为辅。
* **分析**：不要让用户去跟健身房里的职业选手比，要让他们跟“身边的朋友”比。优先展示关注列表的好友排名，点击“查看全网榜”才看到所有用户。

### 四、 界面设计的建议

* **视觉重点（卡片化）**：不要做 Excel 式的表格，要做**荣誉感强的卡片**。
* 第一名：加上金色光芒边框，突出头像。
* 你的名次：**永远固定在底部或顶部显著位置**（即使你没进前十），让用户直观看到“距离前一名还差多少数据”。
* **动作联动**：用户点击榜单上的某人，能直接查看他的“本周训练记录（隐藏具体重量，只看肌群火苗分布）”，这样能引导用户去学习别人的训练计划，增强功能粘性。

### 五、 给你的下一步建议

你现在的系统里已经有了：

1. **动作映射肌肉的逻辑**；
2. **恢复时间的倒计时逻辑**；
3. **本周火苗的记录逻辑**。

**榜单的下一步就是：“用 SQL 聚合这三块逻辑”**。你觉得现有的数据结构，能够直接支撑计算出“本周科学训练得分”吗？如果想做“进步百分比榜”，目前的记录表里是否已经标记了每次训练的重量？



在健身打卡和榜单系统的设计中，**重量数据不能只记录一个简单的数字**。因为在实际训练中，同样是 100kg，做 1 次和做 10 次的含金量完全不同。

为了支撑起一个专业的健身打卡榜单，重量数据的记录通常需要拆解为以下几个维度：

### 1. 基础物理重量（单次记录）

这是最基础的数据，用户在打卡时直接录入：

* **单次负重（Weight）**：本次动作使用的绝对重量（如 60kg）。
* **计重单位（Unit）**：通常为 `kg` 或 `lbs`。*注：在做榜单聚合时，如果系统允许用户切换单位，必须在后台统一换算成 kg 才能进行排名。*
* **自重动作的处理**：如果是引体向上、俯卧撑等自重动作，重量通常记录为 `0`（或者用户自身的体重），榜单此时应转为比拼**次数**。

### 2. 衍生计算重量（榜单排名的核心）

如果你想让榜单更有“含金量”，数据库里需要根据基础重量，自动计算出以下两个核心指标：

#### ① 1RM（单次最大负重估计值）

* **定义**：指用户在某个动作上，**只能规范完成 1 次的极限重量**。
* **为什么需要**：它是健身界公认衡量力量的标准。如果用户 A 卧推 100kg 做了 1 次，用户 B 卧推 80kg 做了 8 次，谁的力量更强？直接比绝对重量不公平。
* **计算公式**：系统可以根据用户的打卡数据（重量 + 次数），利用国际通用的 **Epley 公式** 自动换算：

$$
\text{1RM} = W \times \left(1 + \frac{R}{30}\right)
$$

*(W = 重量，R = 重复次数。例如：80kg 做 8 次，其 1RM 约为 $80 \times (1 + 8/30) = 101.3\text{kg}$)*

* **榜单应用**：“力量巅峰榜”应当以 1RM 作为排行依据，这比单纯看单次重量要科学得多。

#### ② 训练总容量（Volume）

* **定义**：单组重量 $\times$ 次数 $\times$ 组数 的累加值。
* **为什么需要**：这是衡量肌肉肥大（增肌）效果的核心指标。
* **榜单应用**：可以用来做“今日搬砖榜”**或**“本周轰炸榜”。比如某用户今天练腿总共搬了 5 吨的重量，这种累计值对中低重量、高组数的增肌人群极具成就感。

---

### 3. 榜单业务逻辑需要记录的“状态重量”

为了做时间维度的对比（如：本月进步榜），用户的个人数据表中通常需要维护这几个“快照”：

* **历史最大重量（PR - Personal Record）**：用户某动作历史极限（如深蹲 120kg），每次打卡如果超过这个值，系统触发“解锁新纪录”的特效。
* **初始重量（Baseline）**：用户第一次录入该动作时的重量，用来计算“历史总进步幅度”。
* **周期开端重量**：比如本月 1 号的重量，用来计算“本月进步百分比”。

### 📊 数据库设计（伪代码示例）

为了满足上述需求，你的**训练组记录表 (Training_Set_Log)** 大致需要包含这些字段：

```sql
CREATE TABLE training_set_log (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    action_id INT,          -- 动作ID（如：平板卧推）
    weight DECIMAL(5,2),    -- 本组重量 (如 60.00)
    reps INT,               -- 本组次数 (如 10)
    rm_estimate DECIMAL(5,2), -- 自动计算的本组1RM (如 80.00)
    is_pr BOOLEAN,          -- 是否破了个人历史纪录
    created_at TIMESTAMP
);

```

### 💡 针对你的榜单设计的建议

1. **对于力量举爱好者**：做一个“三大项总分榜”（深蹲 1RM + 卧推 1RM + 硬拉 1RM 的总和），这是硬核健身房最受欢迎的榜单。
2. **对于新手**：做一个“容量达人榜”，不拼绝对重量，就拼谁本周累计举起的总重量大（哪怕拿着小哑铃做很多组也能上榜），给新手足够的正向激励。
