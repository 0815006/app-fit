# 📝 会议室预定系统产品需求文档 (PRD)

## 一、 产品概述

本系统是一套专为企业打造的轻量、高效的会议室预定系统，包含微信小程序端（员工用）与 **Web管理端（行政用）**。

* **核心特色**：以 **30分钟** 为最小颗粒度，“即点即约”极速反馈。
* **核心机制**：行政强征具备最高优先级，采用斜纹覆盖告知且支持释放退还；支持最高 8 周的固定周期约及智能后续一键取消。

---

## 二、 微信小程序端（用户端）需求

### 1. 页面整体布局

* **结构**：采用“固定上下，中间滚动”的经典结构。
* **时间边界**：每天纵向时间轴仅展示 **08:00 - 18:00**，18:00 之后整点切断不提供预约。18:00 后员工线下随意使用，系统不做管控。

### 2. 顶部：日期滑动导航（Date Swiper）

* **展示范围**：初始化时展示“今天”，支持横向滑动查看“前 3 天”和“后 3 天”（共 7 天动态区间）。
* **交互规则**：
* 初始化时，“今天”自动居中并高亮显示（加大字号或带品牌底色）。
* 左右滑动切换日期，中部时间轴数据实时无缝刷新。

### 3. 中部：半小时网格时间轴（主看板）

* **颗粒度**：每小时占用一行，内部垂直拆分为两段（上半小时：00-30分；下半小时：30-00分）。
* **全局视图切换按钮（悬浮窗或固定按键）**：
* 采用单键循环切换逻辑：**【预定人视图】（默认） $\rightarrow$ 【会议名称视图】 $\rightarrow$ 回到【预定人视图】**。
* 切换时仅改变格子内文本，不改变格子背景色及底层点击逻辑。
* **格子 4 种状态与视觉规范**：

1. **【空白状态】**：纯白色，代表空闲可预约。
2. **【自己预约】**：填充**颜色 A（如活力蓝）**。根据视图显示“自己的名字”或“会议名称”（若会议名称为空，则显示“已预定”）。
3. **【他人预约】**：填充**颜色 B（如沉稳灰）**。根据视图显示“他人的名字”或“他人会议名称”。
4. **【行政强征】**：渲染为**浅色斜纹条纹（Stripe 纹理，如浅红相间或浅灰蓝相间）**。文字根据视图显示“行政征用”或管理员填写的“征用事由”。

* **过去时间硬性限制**：
* 以格子的**结束时间**与当前手机系统时间对比，凡是已过去的时段，格子做暗色半透明处理。

### 4. 底部：常驻浮窗与详情卡片

* **常驻信息**：常驻屏幕底部，显示当前会议室名称及核心简要标签（如：`会议室 A (301) | 👥12座 | 📺投影仪`）。
* **右侧切换键**：点击弹出公司会议室列表菜单，选中可切换至其他会议室的看板。
* **详情查看**：点击浮窗的信息文字区域，自底向上滑出半屏详情卡片，展示：会议室实景照片、具体楼层位置、精准座位数、配套硬件设施清单（白板、视频会议设备等）。

### 5. 用户点击交互与弹窗逻辑

#### 场景 A：点击【过去时间格子】或【过去的自己预约】

* **表现**：统一为只读。点击过去时间空白处提示“无法预约过去的时间”。点击过去自己/他人的预约，弹出【历史会议详情】，信息全置灰只读，**不提供修改与取消按钮**。

#### 场景 B：点击未来的【空白格子】

1. **即时反馈**：页面格子**立刻变色（颜色 A）并填入自己名字**。
2. **触发弹窗**：随即拉起【补充信息弹窗】，包含：

* 会议名称（输入框，选填）
* 参会人（输入框/选择器，选填）
* **后续按周固定约（开关，默认关闭）**：
* 打开开关后，展开数字输入框 `[请输入周数]`。
* 硬性限制：只能输入整数，**最大限制为 8 周**。

3. **底部按钮**：

* `[ 保存 ]`：保存信息，关闭弹窗。直接点击弹窗外空白处等同于保存。
* `[ 取消预定 ]`：放弃预定，格子恢复纯白。

#### 场景 C：点击未来的【自己预约格子】

* **触发弹窗**：弹出【我的会议详情弹窗】，可编辑修改会议名称和参会人。
* **底部按钮区分**：
* **普通单次预定**：显示 `[ 保存修改 ]` 与 `[ 取消预定 ]`。
* **固定周期约（`is_weekly_fix == true`）**：取消预定按钮分拆为两个：

1. `[ 仅取消当前这一场 ]`：仅释放当前日期的该时段格子。
2. `[ 一键取消后续所有 ]`：**以当前时间点为准，一键删除/释放未来所有关联的后续周会记录**，开过的历史周会保持留痕不动。

#### 场景 D：点击【他人预约格子】或【行政强征（斜纹）格子】

* **触发弹窗**：弹出【只读详情弹窗】。
* 他人预约显示：预定人、会议名称、参会人等。
* 行政强征显示：**征用原因**（如：集团高管战略会）、发起部门。
* **限制**：没有任何输入框与取消按钮，仅有 `[ 我知道了 ]` 关闭键。

## 三、 Web 管理端（后台）需求

### 1. 页面布局

* 采用“左侧主功能菜单，右侧大看板”的标准后台布局。
* **右侧大看板**：横轴为 08:00 - 18:00（半小时一格），纵轴为系统内所有上架的会议室。行政人员可全局鸟瞰任意日期的全公司资源占用。

### 2. 会议室资源管理（生命周期）

* **上架（新增）**：行政可录入会议室名称、具体位置、座位数、勾选配套硬件标签（投影仪/白板/视频设备）、上传实景照片。上架后小程序端实时可见。
* **下架**：一键关闭状态开关，该会议室直接在小程序底部的切换列表中**隐形**。
* **临时维护**：行政可在后台对特定会议室的特定时段发起维护，小程序端对应格子置灰显示“维护中”，拦截用户。

### 3. 最高特权：强制征用与释放

* **征用操作**：行政在大看板上选中任意时段（无论空白还是已被员工预定），右键选择`[强制征用]`。
* **填写征用原因**：必须填写或勾选**征用原因**（用于小程序端斜纹点击展示）。
* **覆盖告知机制（非踢出）**：
* 后台**不删除**原员工的预约记录。
* 小程序端该时段立刻被“浅色斜纹条”覆盖遮挡，原员工同步收到通知。
* **释放退还逻辑**：
* 行政如果在 Web 端取消了本次强制征用，斜纹条瞬间掀开。
* **该时段立刻原封不动地退还给原预定人**（恢复其名字、颜色及会议信息），无需员工重新操作。

## 四、 核心技术埋点与数据落库规则（开发核心）

### 1. 周期约的数据落库方案

当员工在小程序端开启“按周固定约”并输入 $N$ 周（最大8周）后，后端处理逻辑：

* **直接拆分创建**：在数据库的预定表（Booking Table）中，直接一口气生成 $N$ 条独立的记录（每条记录的日期顺延 7 天，时段相同）。
* **特殊字段标记**：这 $N$ 条记录必须被打上以下相同标签：
* `is_weekly_fix = true` （代表是固定约）
* `group_id = UUID()` （同一批生成的记录，共享同一个唯一的组 ID）

### 2. 强征与释放的表设计提示

* 建议采用“占用图层”的设计思路。原预约表不变，新增一张 `Administrative_Lock`（行政征用表）。
* 小程序端渲染时，先拉取预约表，再叠加征用表。如果征用表在该时段有记录，则前端直接渲染为**斜纹图层**并覆盖点击事件；行政一键释放时，直接删除 `Administrative_Lock` 里的记录即可，原预约记录自然重现。
