# 健身打卡小程序（一期）产品需求文档 (PRD)

## 一、 产品概述

- **产品定位**：一款“无脑、低频操作”的健身打卡与肌肉恢复决策工具。
- **核心理念**：不打扰训练。一个动作只操作两次（进场、出场）；不强制记录每组细节，通过用户主观力竭度结合配置，动态生成“肌肉充血与恢复看板”，指导用户“今天该练哪”。
- **核心版本范围**：本周打卡痕迹展示、肌肉状态实时倒计时、动作计时训练、超时修正机制、自由补打卡。

## 二、 核心业务逻辑与算法说明

### 1. 肌肉恢复状态计算（核心核心）

每个肌肉群拥有一个基础恢复时间 $T_{base}$（单位：小时）。用户在结束动作时，滑动选择一个主观力竭度系数 $K$（取值范围：$0.5 \sim 1.2$）。

- **实际所需恢复时间 $T_{real}$**：

  $$
  T_{real} = T_{base} \times K
  $$
- **预计恢复完成时间点 $Time_{target}$**：

  $$
  Time_{target} = 上次训练结束时间(End\_Time) + T_{real}
  $$
- **当下状态判定（跨周延续）**：

  - 若 $当前时间 \ge Time_{target}$，状态 = `🟢 已恢复（Ready）`。
  - 若 $当前时间 < Time_{target}$，状态 = `🟡 恢复中（Recovering）`，并显示剩余倒计时：$Time_{target} - 当前时间$。

### 2. 周期重置逻辑（双轨制）

- **宏观（本周打卡痕迹）**：以自然周（周一 00:00 至周日 24:00）为单位。周一 00:00 准时清零所有肌群的“本周打卡次数”。
- **微观（实时恢复状态）**：不随周一重置。如果周日晚上练了腿，周一早上依然根据算法计算并展示 `🟡 恢复中`。

## 三、 功能需求与页面交互详情

### 1. 页面一：首页（本周训练看板）

- **用户场景**：用户进入小程序，一眼看出这周练得够不够，以及今天当下哪些肌肉恢复好了可以练。
- **界面元素**：
  - **顶部状态提示**：一句话文案，如：“今天适合训练的部位：背部、二头肌”。
  - **肌群卡片列表**：平铺展示全身核心肌群（如：胸、背、腿、肩、二头、三头、腹）。
- **卡片内部交互与视觉**：
  - **左侧（本周宏观痕迹）**：展示本周该肌群打卡过的动作次数（例如：🔥 $\times$ 3）。周一 0点清零。
  - **右侧（当下微观状态）**：
    - 状态 A（未练过或已回满血）：显示 `🟢 已恢复`，卡片高亮。
    - 状态 B（正在恢复中）：显示 `🟡 恢复中`，并动态倒计时（形如：`还剩 14 小时`），卡片半透明变灰。
  - **点击卡片**：进入【页面二：动作选择页】。允许用户多选肌群（例如勾选“胸”和“三头”），点击底部大按钮“进入动作选择”。

### 2. 页面二：动作选择与指导页

- **用户场景**：用户决定了今天要练的肌群，进入查看该肌群下的动作和推荐配置，并随时可以取消或换动作。
- **界面元素**：
  - 列表展示上一步所选肌群关联的所有动作及器械。
  - 每个动作卡片展示：动作名称、关联器械、**纯文本指导建议**（推荐组数、一组几个、推荐组间休息时间）。
- **页面交互**：
  - **临时更换/取消**：用户可点击左上角返回或取消选择。
  - **开始训练**：点击某动作卡片上的【开始训练】按钮，立刻进入【页面三】。

### 3. 页面三：动作训练进行页（核心计时页）

- **用户场景**：手机丢在一边专心训练，不强制要求每组练完去点手机，只展示计时和指导。
- **界面元素**：
  - **大码表计时器**：居中展示，从 `00:00` 开始正计时，记录该动作总训练时长。
  - **纯文本指导看板**：置于计时器下方，静态展示：推荐 $4$ 组 $\times$ $10$ 次、组间休息建议 $90$ 秒、动作要领。
- **页面交互**：
  - **【结束训练】按钮**：点击时**必须触发二次确认弹窗**（“确定要结束本次动作训练吗？”），防止汗水误触。
  - **【更换动作/取消】按钮**：点击清空当前计时，返回页面二。

### 4. 页面四：结束评价页（力竭度滑动）

- **用户场景**：练完一个动作，拿起手机盲操滑动，评估本次力竭程度，一键收工。
- **界面元素**：
  - **主观力竭度滑块（Slider）**：取值范围 $50\% \sim 120\%$。
    - 左端（$50\%$）：浅练/浅尝辄止。
    - 中间（$100\%$）：刚刚好力竭/完美达成。
    - 右端（$120\%$）：超负荷轰炸/彻底练废。
  - **【确认提交】大按钮**：点击后更新数据库，刷新该肌群最新训练时间，自动跳回首页。

### 5. 全局机制：超时修正弹窗

- **触发条件**：用户重新打开小程序进入首页时，后端校验该用户是否存在 `status = 训练中` 且 `当前时间 - start_time > 2小时` 的异常记录。
- **交互表现**：直接在首页上方弹出不可强制关闭的**拦截式弹窗**。
  - **文案**：“检测到你于 [昨天 18:30] 开始了 [杠铃卧推]，可能忘记点结束了。”
  - **用户修正输入**：
    1. 输入/滑动选择本次动作实际练了多少分钟（默认预设值：$15$ 分钟）。
    2. 滑动选择当时的力竭程度。
  - **点击【修正并结束】**：后端自动将 `end_time` 修正为 `start_time + 用户输入的分钟数`，并以此重新计算肌肉恢复倒计时，关闭弹窗。

### 6. 辅助功能：自由补打卡

- **入口位置**：首页右上角【自由补卡】。
- **用户场景**：出了健身房或第二天，随时补记忘记打卡的动作。
- **交互流程**：
  1. 日期时间选择器（允许自由选择过去任何一天的某个时间点）。
  2. 选择肌群 -> 选择动作。
  3. 滑动力竭度。
  4. 点击提交。
- **覆盖规则**：补卡记录正常写入历史，但首页看板计算动态恢复时间时，**永远只抓取该肌群离当前时间最近（时间戳最大）的一条记录**进行倒计时推算。

## 四、 数据库表结构设计建议（技术底座追加）

在一期开发中，你需要在原有的元数据表（动作、肌群、器械）基础上，至少追加以下两张业务表：

### 1. 训练记录总表 (`gym_workout_record`)

用于记录每一次动作训练的生命周期。

| **字段名**     | **类型** | **说明**                                            |
| -------------------- | -------------- | --------------------------------------------------------- |
| `id`               | VARCHAR(32) (PK)    | 主键 ID（雪花ID）                                                   |
| `user_id`          | VARCHAR(32)    | 所属用户ID（关联 `user.id`）                                             |
| `action_id`        | VARCHAR(32)            | 关联的动作 ID                                             |
| `muscle_group`     | Varchar        | 冗余字段：所属肌群大类编码（方便首页直接 Group By 查询）          |
| `start_time`       | DateTime       | 点击开始训练的时间戳                                      |
| `end_time`         | DateTime       | 点击结束（或经超时修正后）的时间戳                        |
| `exhaustion_score` | DECIMAL(3,2)          | 用户滑动的力竭度系数（0.5 ~ 1.2）                         |
| `status`           | TinyInt        | 状态：`0: 训练中`, `1: 正常结束`, `2: 超时修正结束` |

### 2. 肌肉基础配置表 (`muscle_group_config`)

用于配置每个肌肉群天生的恢复时间。

| **字段名**        | **类型** | **说明**                           |
| ----------------------- | -------------- | ---------------------------------------- |
| `muscle_group`        | Varchar (PK)   | 肌群唯一标识（如 'chest', 'back'）       |
| `base_recovery_hours` | Int            | 基础恢复小时数（如胸配置 48，腿配置 72） |

这份 PRD 将极简主义和健身实战场景紧密结合。你可以直接以此为标准开始后端 API 接口设计和前端排版了！下一步，需要为你梳理前端组件的开发优先级，还是直接开始设计数据库查询 SQL 语句？

## 表2挪到肌肉群表中

完全可以，而且这样设计更精简！直接把 `base_recovery_hours`（基础恢复小时数）作为字段加到现有的**肌肉群表**中，可以少维护一张表，多表关联查询（Join）也省了。

这样调整后，你一期的数据库业务部分就只需要追加**一张核心表**了：

### 1. 肌肉群表（你原有的元数据表追加字段）

在原有的肌肉群表里，直接补上基础恢复时间：

- `id` / `muscle_code` (如 'chest')
- `muscle_name` (如 '胸肌')
- 👉 **`base_recovery_hours` (Int)**：基础恢复小时数（大肌群配 72/48，小肌群配 24）

### 2. 训练记录表（新增的唯一业务表 `gym_workout_record`）

记录每次动作的打卡流水：

- `id`（雪花ID）
- `user_id`（关联 `user.id`，标识哪位用户）
- `action_id`（关联动作）
- `muscle_group_code`（关联肌肉群，用来直接找肌肉表里的基础恢复时间）
- `start_time`（开始时间戳）
- `end_time`（结束时间戳）
- `exhaustion_score`（力竭度系数 0.5 ~ 1.2）
- `status`（0:训练中，1:正常结束，2:超时修正结束）

### 💡 这样设计后，首页看板的查询逻辑（SQL思路）

当用户进入首页时，后端只需要查出**该用户每个肌群最后一次（`max(end_time)`）的训练记录**，然后和肌肉表里的 `base_recovery_hours` 算一下就行了。

伪代码逻辑大概是：

1. **查出本周打卡次数**：`SELECT count(1) FROM gym_workout_record WHERE user_id = ? AND muscle_group_code = 'chest' AND start_time >= 本周一0点`。
2. **查出最后一次训练算恢复**：`SELECT end_time, exhaustion_score FROM gym_workout_record WHERE user_id = ? AND muscle_group_code = 'chest' ORDER BY start_time DESC LIMIT 1`。
3. 用这条最新记录的 `end_time + (base_recovery_hours * exhaustion_score)`，跟 `当前时间` 对比，直接就能在首页给"胸肌"挂上 `🟢已恢复` 还是 `🟡恢复中（倒计时）` 的标签。

表结构这样合并后非常干净！你现在后端和数据表思路已经全通了，打算先去写后端的这几个核心接口（开始训练、结束训练、首页看板），还是开始搭微信小程序的前端页面？
