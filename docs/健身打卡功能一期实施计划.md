## 一、数据库变更（Flyway）

### 1.1 Flyway V18: `gym_muscle` 追加 `base_recovery_hours` 字段

在现有 [`gym_muscle`](../java-fit-server/src/main/resources/db/migration/V4__create_gym_muscle_table.sql) 表上追加字段：

```sql
ALTER TABLE gym_muscle
  ADD COLUMN base_recovery_hours INT NOT NULL DEFAULT 48
  COMMENT '基础恢复小时数（大肌群72/48，小肌群24）';
```

然后按肌群大类批量更新，确保同一 `muscle_group` 下所有肌群的 `base_recovery_hours` 值一致：

```sql
UPDATE gym_muscle SET base_recovery_hours = 48 WHERE muscle_group = 'CHEST';
UPDATE gym_muscle SET base_recovery_hours = 48 WHERE muscle_group = 'BACK';
UPDATE gym_muscle SET base_recovery_hours = 24 WHERE muscle_group = 'SHOULDER';
UPDATE gym_muscle SET base_recovery_hours = 24 WHERE muscle_group = 'ARM';
UPDATE gym_muscle SET base_recovery_hours = 72 WHERE muscle_group = 'LEG';
UPDATE gym_muscle SET base_recovery_hours = 48 WHERE muscle_group = 'GLUTE';
UPDATE gym_muscle SET base_recovery_hours = 24 WHERE muscle_group = 'CORE';
```

> **注意**：`gym_muscle` 表记录的是**具体肌肉**（如胸大肌、背阔肌），此处 `base_recovery_hours` 按 `muscle_group`（肌群大类）维度维护。看板查询时通过 `muscle_group` 分组，取任一条记录的 `base_recovery_hours` 即可（同一 group 下值一致）。

> **命名说明**：不新建 `muscle_group_config` 表，直接沿 PRD 末尾的建议合并到 `gym_muscle` 中。

### 1.2 Flyway V19: 新建核心业务表 `gym_workout_record`

```sql
CREATE TABLE gym_workout_record (
    id               VARCHAR(32)   NOT NULL COMMENT '主键ID（雪花ID）',
    emp_no           VARCHAR(7)    NOT NULL COMMENT '员工工号（7位）',
    action_id        VARCHAR(32)   NOT NULL COMMENT '动作ID（关联 gym_action.id）',
    muscle_group     VARCHAR(50)   NOT NULL COMMENT '冗余：肌群大类编码（CHEST/BACK/SHOULDER/ARM/LEG/GLUTE/CORE 等）',
    start_time       DATETIME      NOT NULL COMMENT '训练开始时间',
    end_time         DATETIME      DEFAULT NULL COMMENT '训练结束时间（结束时/修正时写入）',
    exhaustion_score DECIMAL(3,2)  DEFAULT NULL COMMENT '力竭度系数（0.50 ~ 1.20）',
    status           TINYINT       NOT NULL DEFAULT 0 COMMENT '状态：0=训练中, 1=正常结束, 2=超时修正结束',
    create_time      DATETIME      DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time      DATETIME      DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (id),
    KEY idx_emp_no (emp_no),
    KEY idx_action_id (action_id),
    KEY idx_muscle_group (muscle_group),
    KEY idx_status (status),
    KEY idx_start_time (start_time),
    KEY idx_emp_muscle_time (emp_no, muscle_group, start_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='健身打卡训练记录';

```

> **命名说明**：PRD 中叫 `workout_record`，实际使用 `gym_workout_record` 以与现有 `gym_*` 系列表命名一致。`muscle_group` 字段存的是**肌群大类编码**（如 `CHEST`），对应 [`gym_muscle.muscle_group`](../java-fit-server/src/main/resources/db/migration/V4__create_gym_muscle_table.sql:8)。

### 2.3 Service 层

#### 新建 `GymWorkoutRecordService` 接口 + `impl/GymWorkoutRecordServiceImpl`

核心方法：

| 方法 | 说明 |
|------|------|
| `startWorkout(String actionId)` | 1. 通过 `gym_action_muscle_rel`（取 `is_primary=1`）查到 `muscle_id`；2. 通过 `gym_muscle` 查到 `muscle_group`；3. 创建记录（status=0），返回 recordId |
| `endWorkout(String recordId, BigDecimal exhaustionScore)` | 写入 `end_time = now`, `exhaustion_score`, status→1 |
| `correctTimeout(String recordId, int actualMinutes, BigDecimal exhaustionScore)` | `end_time = start_time + actualMinutes` 分钟，status→2 |
| `getDashboard(String empNo)` | **核心看板查询**：返回所有肌群的恢复状态 + 本周打卡次数 |
| `makeupWorkout(String actionId, LocalDateTime startTime, BigDecimal exhaustionScore)` | 自由补打卡：`end_time = startTime`（即时刻打卡，无时长），status=1 |
| `checkTimeout(String empNo)` | 检查 status=0 且 `now - start_time > 2小时` 的记录 |

#### `getDashboard` 逻辑（与 PRD 算法对齐）：

1. 查所有肌群（`gym_muscle` 全表），按 `muscle_group` 分组，每组取第一条的 `base_recovery_hours`
2. 对每个肌群大类（`muscle_group`），查：
   - **本周打卡次数**（PRD："本周该肌群打卡过的动作次数"）：
     `SELECT COUNT(DISTINCT action_id) FROM gym_workout_record WHERE emp_no=? AND muscle_group=? AND start_time >= 本周一00:00`
   - **最新训练记录**（PRD："永远只抓取该肌群离当前时间最近（时间戳最大）的一条记录"）：
     `SELECT end_time, exhaustion_score FROM gym_workout_record WHERE emp_no=? AND muscle_group=? AND status IN (1,2) ORDER BY end_time DESC LIMIT 1`
3. 恢复状态计算（PRD 公式）：
   - `T_real = base_recovery_hours × exhaustion_score`（小时）
   - `Time_target = end_time + T_real`
   - 若 `now >= Time_target` → `READY`（🟢 已恢复）
   - 若 `now < Time_target` → `RECOVERING`（🟡 恢复中），`remaining_seconds = Time_target - now`
4. **顶部提示文案**：汇总所有 `READY` 状态的肌群名称，前端拼接为「今天适合训练的部位：背部、二头肌」

### 2.5 API 接口清单

| 方法 | 路径 | 请求体 | 说明 |
|------|------|--------|------|
| `POST` | `/api/gym-workout/start` | `{ actionId }` | 开始训练。后端内部通过 `actionId → gym_action_muscle_rel(is_primary=1) → gym_muscle.muscle_group` 自动推导肌群大类 |
| `PUT` | `/api/gym-workout/{id}/end` | `{ exhaustionScore }` | 结束训练，写入 end_time=now + 力竭度 |
| `PUT` | `/api/gym-workout/{id}/correct` | `{ actualMinutes, exhaustionScore }` | 超时修正，end_time = start_time + actualMinutes |
| `GET` | `/api/gym-workout/dashboard` | — | 获取首页看板（含 readyMuscleNames 顶部提示 + 肌群列表 + 超时记录） |
| `POST` | `/api/gym-workout/makeup` | `{ actionId, startTime, exhaustionScore }` | 自由补打卡，end_time = startTime（瞬间打卡） |
| `GET` | `/api/gym-workout/timeout-check` | — | 检查本用户是否有 status=0 且超 2h 的记录 |

### 2.6 DTO / VO

| 类型 | 类名 | 字段 |
|------|------|------|
| DTO | `StartWorkoutDTO` | `actionId: String` |
| DTO | `EndWorkoutDTO` | `exhaustionScore: BigDecimal` |
| DTO | `CorrectTimeoutDTO` | `actualMinutes: int, exhaustionScore: BigDecimal` |

## 六、关键设计决策

1. **表名**：PRD 中的 `workout_record` → `gym_workout_record`（与现有 `gym_` 前缀统一）
2. **`base_recovery_hours`**：PRD 建议独立建表 `muscle_group_config`，但按 PRD 末尾补充说明，合并到 `gym_muscle` 表更简洁
3. **Web 端交互模式**：不走多页面跳转，用单页面 + 弹窗（Dialog）模式，适配桌面大屏体验
4. **小程序端**：保持原生多页面跳转模式（navigateTo），符合微信小程序用户习惯
5. **共用后端接口**：Web 和小程序完全共用同一套 `/api/gym-workout` 接口，仅前端交互不同
